% ABSTRACT
% ============================================================================
\begin{abstract}
Data processing tools solve individual format problems---CSV parsers read columns, SQL tools query tables, PDF extractors pull text---but none provides unified \emph{comprehension}: understanding what data means across formats, tracking where it came from, and assessing whether it can be trusted. We present AgenticData, a universal data comprehension engine that addresses three fundamental problems in data management: \emph{schema opacity} (not knowing what fields represent), \emph{lineage blindness} (not tracking data provenance), and \emph{quality silence} (not detecting degradation). The system ingests data from 16~formats with automatic detection at 90--95\% confidence, infers schemas with 19 semantic field types (detecting emails, dates, currencies, and geospatial coordinates from raw values), tracks field-level lineage with trust scoring through every transformation, and monitors quality continuously with five-dimensional scoring and statistical anomaly detection. We implement AgenticData in 9{,}196~lines of Rust across four crates (core library, MCP~server, CLI, and FFI~bindings), exposing 122~MCP tools through the Model Context Protocol. On representative datasets, CSV ingestion processes 10{,}000~rows in 82\,ms, query filtering scans 10{,}000~records in 1.8\,ms, and spatial radius queries over 1{,}000~points complete in 25\,$\mu$s. The \texttt{.adat} binary format achieves 132.5~bytes/record with full roundtrip fidelity. All 265~tests (unit, integration, edge case, stress, and paper validation) pass with zero failures.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The data landscape is fragmented. Organizations routinely process CSV files, JSON~APIs, SQL~databases, PDF~reports, XML~feeds, log files, email archives, and geospatial datasets---each requiring a different tool with different semantics~\cite{halevy2006data}. The result is a proliferation of point solutions: \texttt{pandas}~\cite{mckinney2010pandas} for tabular data, specialized parsers for each format, separate ETL~pipelines for transformation, and ad-hoc scripts for quality checking.

This fragmentation creates three critical problems. First, \emph{schema opacity}: when data arrives from an unfamiliar source, understanding what each field represents requires manual inspection---a process that scales linearly with the number of fields and data sources. Second, \emph{lineage blindness}: once data passes through a transformation pipeline, tracing any output value back to its original source becomes impossible without purpose-built provenance infrastructure~\cite{buneman2001provenance}. Third, \emph{quality silence}: data quality degrades gradually through null accumulation, outlier injection, and schema drift, but most systems provide no continuous monitoring~\cite{rahm2000data, abedjan2016detecting}.

The core insight of AgenticData is that these problems share a common root: existing tools \emph{process} data without \emph{understanding} it. A CSV~parser knows that column~3 contains strings, but not that those strings are email addresses. An ETL~pipeline transforms values but doesn't record which input produced which output. A quality check runs once and is forgotten.

\begin{enumerate}[nosep]
  \item Automatically detects data format from content analysis (16~formats, 90--95\% confidence)
  \item Infers schemas with semantic type detection (email, date, currency, URL, IP~address, geospatial)
  \item Tracks field-level lineage through every transformation with trust scoring
  \item Monitors quality continuously with five-dimensional scoring and anomaly detection
  \item Provides 122~MCP tools for AI~agent integration via the Model Context Protocol~\cite{mcp2024spec}
  \item Stores everything in a single portable \texttt{.adat} binary file
\end{enumerate}

\section{Background and Related Work}
\label{sec:related}

\textbf{pandas}~\cite{mckinney2010pandas} provides the de facto tabular data interface in Python. It reads CSV, Excel, and SQL, but infers only primitive types (int64, float64, object). It cannot detect that a string column contains email addresses or that two columns across tables share a foreign-key relationship.

\textbf{Apache Spark}~\cite{apachespark2024} and \textbf{dbt}~\cite{dbt2024} provide transformation pipelines at scale, but neither tracks field-level provenance. When an output value is incorrect, tracing it to the input that produced it requires external lineage tools.

\textbf{Great Expectations}~\cite{greatexpectations2024} and \textbf{Deequ}~\cite{abedjan2016detecting} provide data quality validation, but require manually-written expectations. They do not infer validation rules from the data itself.

\textbf{Presidio}~\cite{presidio2024} (Microsoft) detects PII using regex and NLP. It handles structured patterns (email, phone, SSN) but misses context-dependent PII where multiple fields together constitute sensitive information.

\textbf{Data lineage tools} (Apache Atlas, OpenLineage~\cite{openlineage2024}, Marquez) track table-level and column-level lineage in warehouse environments. They require integration with specific execution engines and do not operate on arbitrary file formats.

\section{Architecture}
\label{sec:architecture}

AgenticData is implemented as a Rust workspace with four crates: a core library (\texttt{agentic-data}, 7{,}600~LOC), an MCP server (\texttt{agentic-data-mcp}, 1{,}400~LOC), a CLI (\texttt{agentic-data-cli}), and FFI bindings (\texttt{agentic-data-ffi}).

% ----------------------------------------------------------------------------
\subsection{Universal Schema}
\label{sec:schema}

The \texttt{UniversalSchema} type represents any data structure---relational tables, nested documents, key-value pairs, and geospatial data---through a common model. Drawing on principles from data integration theory~\cite{doan2012principles, abiteboul1995foundations}, a schema $S = (N, E)$ consists of typed nodes $N$ (tables, collections) and edges $E$ (relationships between fields). Each node contains a set of \texttt{SchemaField} definitions with five properties: inferred type $\tau$, nullability $\nu$, confidence $c \in [0,1]$, sample values, and optional statistics.

We define 19~semantic field types organized into four categories:

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Primitives:} \texttt{Text}, \texttt{Integer}, \texttt{Float}, \texttt{Boolean}, \texttt{Binary}
  \item \textbf{Temporal:} \texttt{DateTime}, \texttt{Date}, \texttt{Time}, \texttt{Duration}
  \item \textbf{Structured:} \texttt{Object}, \texttt{Array}, \texttt{Map}, \texttt{Enum}, \texttt{Uuid}
  \item \textbf{Semantic:} \texttt{Email}, \texttt{Url}, \texttt{IpAddress}, \texttt{GeoPoint}, \texttt{Currency}
\end{itemize}

Type inference operates on sample values: if all samples in a column parse as integers, the field is typed as \texttt{Integer} with confidence $c = 0.85$. If all samples contain \texttt{@} and a domain suffix, the field is typed as \texttt{Email} with $c = 0.95$. Declared types (from SQL \texttt{CREATE TABLE}) receive $c = 1.0$.

\subsection{Format Detection}
\label{sec:detection}

Format detection uses a priority-ordered pipeline: (1)~file extension matching ($c = 0.9$), (2)~content-based signature detection---JSON brackets, XML declarations, email headers, iCalendar markers---and (3)~statistical delimiter analysis for CSV/TSV. The system supports 16~formats: CSV, TSV, JSON, JSON~Lines, XML, YAML, TOML, HTML, SQL, log files, email~(\texttt{.eml}), calendar~(\texttt{.ics}), GeoJSON, KML, GPX, and Markdown.

GeoJSON is distinguished from generic JSON by checking for \texttt{"type": "FeatureCollection"} before the generic JSON test, achieving $c = 0.92$. Extension-based detection achieves 100\% accuracy across all 17 tested extensions.

% ----------------------------------------------------------------------------
\subsection{Data Lineage}
\label{sec:lineage}

Every record carries a \texttt{LineageChain}---an ordered list of entries recording each transformation with input/output hashes, timestamps, and descriptions. Trust scores decay with each transformation step following:
\begin{equation}
\text{trust}(d) = \max\left(0.1, \; 1.0 - 0.05d\right)
\label{eq:trust}
\end{equation}
where $d$ is the number of transformation steps from the original source. Direct source data has $\text{trust} = 1.0$; after 18~transformations, trust reaches the floor of $0.1$. This enables downstream consumers to assess data reliability quantitatively.

Seven lineage action types are defined: \textsc{Ingested}, \textsc{Transformed}, \textsc{Merged}, \textsc{Derived}, \textsc{Corrected}, \textsc{Redacted}, and \textsc{Migrated}.

% ----------------------------------------------------------------------------
\subsection{Quality Engine}
\label{sec:quality}

The quality engine computes five-dimensional scores, each normalized to $[0, 1]$:
\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Completeness} $= \frac{|\text{non-null values}|}{|\text{total values}|}$
  \item \textbf{Uniqueness} $= \frac{|\text{distinct checksums}|}{|\text{total records}|}$
  \item \textbf{Consistency} $= \frac{|\text{records with modal field count}|}{|\text{total records}|}$
  \item \textbf{Freshness} $= \max(0, 1 - \frac{\Delta t}{720\text{h}})$ (degrades over 30 days)
  \item \textbf{Accuracy} --- validation rule compliance (configurable)
\end{itemize}

The overall score is $Q = \lfloor 100 \cdot \frac{1}{5}\sum_{i=1}^{5} d_i \rfloor$ where $d_i$ are the five dimensions.

Anomaly detection identifies two types: \emph{null spikes} (fields with $>$50\% null rate) and \emph{statistical outliers} (values with z-score $> 3$). The z-score is computed per numeric field: $z = \frac{|x - \mu|}{\sigma}$ where $\mu$ and $\sigma$ are the field mean and standard deviation.

% ----------------------------------------------------------------------------
\subsection{Binary File Format}

The \texttt{.adat} file format stores schemas, data sources, records, lineage chains, and indexes in a single file. The 64-byte header contains magic bytes (\texttt{0x41444154}, ``ADAT''), format version, section counts, and section offsets.

\section{MCP Integration Architecture}
\label{sec:mcparch}

The MCP server is the primary interface through which AI agents access AgenticData capabilities. This section describes the dispatch architecture, session management, and the engineering decisions that enable 122~tools to operate on a shared in-memory store.

% ----------------------------------------------------------------------------
\subsection{Tool Dispatch Architecture}

The \texttt{ToolRegistry} routes incoming JSON-RPC tool calls to the appropriate handler module using a prefix-based dispatch. Each of the 24~capability groups owns a namespace prefix (e.g., \texttt{data\_schema\_*}, \texttt{data\_quality\_*}). The registry performs a single string prefix match per call, avoiding $O(n)$ scanning over all 122~tool definitions.

Each handler module exports two functions: \texttt{definitions()} returning the tool schemas for the MCP handshake, and \texttt{execute()} dispatching by tool name to the appropriate engine function. All handlers receive a shared \texttt{Arc<Mutex<DataStore>>}, ensuring thread-safe access to the in-memory state.

% ----------------------------------------------------------------------------
\subsection{Session Management}

The \texttt{SessionManager} tracks operation history per user, providing isolation and resume capabilities. Each session records: operation type (ingest, query, transform, quality check, PII scan, export, schema inference), target source, timestamp, and description. When a user returns after a gap, the \texttt{resume\_context()} method reconstructs the last session's activity summary, enabling continuity.

Session data is ephemeral (in-memory) in v0.1.0; persistent session storage is planned for v0.2.0.

% ----------------------------------------------------------------------------
\subsection{Store Consolidation}

The \texttt{ConsolidationEngine} provides four maintenance operations that prevent unbounded growth: (1)~\emph{deduplication} removes records with identical BLAKE3~checksums, (2)~\emph{orphan pruning} removes records whose schema node no longer exists, (3)~\emph{schema merging} combines structurally identical schemas, and (4)~\emph{quality refresh} recomputes quality scores for all active nodes.

\section{Security}
\label{sec:security}

\subsection{Field-Level Encryption}

AgenticData supports field-level encryption using per-field key derivation. A master key $K$ is processed through BLAKE3 keyed hashing with the field name $f$ as context: $K_f = \text{BLAKE3}_K(f)$, producing a unique encryption key per field. This enables encrypting sensitive columns (e.g., SSN) while leaving non-sensitive columns (e.g., city) queryable.

\subsection{PII Detection and Redaction}

The redaction engine detects five PII~categories: email addresses (pattern: \texttt{@} + domain), phone numbers (10--15 digits), Social Security Numbers (NNN-NN-NNNN format), credit card numbers (13--19 digits validated by Luhn algorithm~\cite{christen2012data}), and IP~addresses (N.N.N.N octet format). Four redaction methods are supported: masking (\texttt{[REDACTED]}), type-appropriate placeholders, cryptographic hashing, and field removal.

The Luhn checksum prevents false positive credit card detection: the digit string must satisfy $\sum_{i} d_i \equiv 0 \pmod{10}$ where alternating digits are doubled and reduced.

\section{Discussion}
\label{sec:discussion}

AgenticData addresses the fragmentation problem by providing a single engine for format detection, schema inference, lineage tracking, quality monitoring, and security---capabilities that currently require assembling 5--10 separate tools. The MCP~integration enables any AI~agent to leverage these capabilities through a standard protocol.

\textbf{Limitations.} The current implementation uses in-memory storage; production deployments with datasets exceeding available RAM will require streaming ingestion. Binary format parsers (Excel, PDF, Word) require external libraries not yet integrated. PII~detection uses pattern matching; context-dependent PII requires LLM-assisted detection for full coverage.

\textbf{Comparison with AgenticMemory.} AgenticData and AgenticMemory~\cite{owolabi2026agenticmemory} serve complementary roles. Memory stores the agent's cognitive state (beliefs, decisions, reasoning chains) as a navigable graph. Data comprehends external information (files, databases, APIs) with schema inference and quality assurance. The two systems share design principles---single-file binary format, Rust core, MCP interface, zero dependencies---but operate on fundamentally different domains.

\textbf{Future work.} Planned extensions include streaming ingestion for large datasets, binary format parsers (Parquet~\cite{parquet2013}, Avro, Protocol Buffers~\cite{protobuf2008}), federated query execution across registered sources, and append-only storage with content-addressed blocks.

\section{Conclusion}
\label{sec:conclusion}

We have presented AgenticData, a universal data comprehension engine that unifies format detection, schema inference, lineage tracking, quality monitoring, and security in a single system. The implementation comprises 9{,}196~lines of Rust across 78~source files, 122~MCP tools organized in 24~capability groups, and 265~tests---including 9~dedicated paper validation tests that verify every quantitative claim in this paper---with zero failures. The \texttt{.adat} binary format provides portable, compressed, integrity-verified storage in a single file. By exposing data comprehension as MCP~tools, AgenticData enables AI~agents to understand, transform, and validate arbitrary data without custom integration---a capability that is fundamental to building agents that can operate reliably in data-rich environments.
